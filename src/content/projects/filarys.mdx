## Problem space

The Italian agri-food industry is fragmented across dozens of disconnected software tools: farm management software, logistics platforms, ERP systems, quality-control dashboards, and compliance registries rarely talk to each other. Producers, cooperatives, logistics operators, and distributors each operate in their own silo, making end-to-end traceability nearly impossible and EU regulatory compliance a manual, error-prone process.

Filarys is designed to replace this fragmentation with a single, modular platform that covers the entire supply chain — from field to shelf — with a shared data model, real-time event flow, and built-in compliance logic.

## Target users

Supply chain participants are modelled as **organisations** with an explicit role:

- **Primary** — farms, cooperatives: field management, input tracking, yield recording
- **Secondary** — processors, packagers: lot tracking, quality control, transformation records
- **Tertiary** — logistics operators, distributors, retailers: transport scheduling, delivery confirmation, shelf traceability

Each organisation is an independent tenant on the platform. A cooperative and a logistics company can share traceability data without ever sharing a database row.

## Architecture

The platform is built on a **Clean Architecture + CQRS** foundation using **.NET 9** and **MediatR**. Every business operation is modelled as an explicit command or query, keeping domain logic fully decoupled from infrastructure concerns.

**Multi-tenancy** is implemented at the PostgreSQL session level. A custom EF Core `DbConnectionInterceptor` sets a session-scoped variable on every connection open, which Row-Level Security policies then use to filter all queries automatically. Tenant isolation is enforced at the database engine — not in application code — so no query can accidentally leak cross-tenant data.

**Tenant resolution** happens in a dedicated ASP.NET Core middleware that runs after authentication. The middleware extracts the tenant identifier from the authenticated JWT and populates a scoped `ITenantContext`. A header-based fallback is available in development only and is stripped in production configuration.

**Modular registration** uses an `IModelCreatingContributor` interface: each module contributes its own EF Core entity configurations independently. This allows modules to be added or removed without touching the shared `DbContext`.

**Cross-cutting infrastructure** is provided by an internal shared library (`StangaNetLib`) covering: structured observability (correlation IDs, request logging, security headers), distributed caching, rate limiting, idempotency key deduplication, JWT authentication, media storage, and GDPR consent middleware. Modules opt in by calling the relevant extension method in their `DependencyInjection` class.

## Modules

The backend is structured as a set of independently deployable modules, each owning its own domain, persistence configuration, and API surface.

| Module | State | Scope |
|---|---|---|
| **Tenancy** | Active | Organisation lifecycle (create, suspend, exit), supply chain role assignment, tenant context propagation |
| **Identity** | Active | OIDC/JWT authentication wired to the shared auth library; token validation and revocation |
| **Tokens** | Scaffold | Token issuance and management (domain design complete) |
| **AccessGrants** | Scaffold | Cross-tenant data sharing agreements (domain design complete) |
| **Sectors** | Scaffold | Agri-food sector classification and regulatory zone mapping |
| **Assets** | Scaffold | Document and media storage backed by `StangaNetLib.Media` |
| **Import** | Scaffold | Structured data ingestion from external sources |
| **Export** | Scaffold | Report generation and data export (PDF, XML) |
| **PublicView** | Scaffold | Consumer-facing traceability endpoints (QR code scan flow) |
| **Audit** | Scaffold | Immutable audit log for all domain events |
| **GDPR** | Active | Consent management infrastructure wired to `StangaNetLib.Gdpr`; consent repository pluggable per deployment |

## Development approach

Filarys is a solo project following a structured, documentation-first methodology:

1. **Domain modelling first** — bounded context map, aggregate design, and ubiquitous language are fully documented before implementation begins, to avoid costly late-stage refactors.
2. **Legal and regulatory review** — Italian agricultural law (D.Lgs. 99/2004, EU Reg. 2018/848), GDPR data residency requirements, and traceability evidentiary standards are reviewed before the data model is finalised.
3. **Test suites alongside spec** — acceptance criteria and integration test skeletons are written at design time, not after implementation.
4. **Dependency-ordered delivery** — Tenancy and Identity ship first (they underpin every other module); domain modules follow once the isolation layer is proven stable.

## Current status

The core infrastructure layer is fully implemented and tested: multi-tenant PostgreSQL isolation, middleware pipeline, module registration pattern, and cross-cutting concerns. The Tenancy module has a complete domain model (Organisation aggregate with domain events, lifecycle transitions, and supply chain role assignment).

Remaining modules are structural scaffolding — the registration wiring, project references, and test harness are in place; domain logic implementation is the active next phase.

Parallel tracks currently open:
- **Regulatory mapping** — translating PAC/PSR rule sets and Nitrati directive zones into enforceable domain invariants
- **AccessGrants domain design** — modelling cross-tenant data sharing as a first-class aggregate
- **Integration test suite** — writing scenario coverage for the Tenancy and Identity modules before expanding to others
